# プロジェクトゴール — 詳細実装計画の策定可能性評価

## 本文書の目的

対話セッション（プロンプト）および作成済みの `planning_doc.md` を入力として、
**実装の詳細レベルの計画書を作成できるか**を評価する。

各実装コンポーネントについて「定義済み / 方針のみ / 未定義」を判定し、
詳細計画に進むための準備状況と残課題を明らかにする。

---

## 前提条件

### ライブラリ提供として省略するコンポーネント

以下は外部ライブラリ/インフラとして「使えるシステム」が存在する前提とし、
本プロジェクト内での設計・実装は行わない。

| コンポーネント | 省略理由 | 本プロジェクトとの接点 |
|---|---|---|
| **recall システム** | ライブラリ提供 | 各アクターの記憶読み書きAPIとして利用。情報源制限ルール（セクション8.1）に従うラッパーのみ自前実装 |
| **タイムスケール管理** | ライブラリ提供 | アクターの活性/休眠/待機の状態遷移エンジンとして利用。活性化判定ロジック（セクション4.3）のみ自前実装 |
| **ハートビート実装** | ライブラリ提供 | アクターの定期更新スケジューリングとして利用。更新頻度パラメータの設定のみ自前 |

## 1. コンポーネント別 実装準備状況の評価

### 評価基準

```
◎ 実装可能: 計画書に十分な仕様があり、詳細設計→実装に進める
○ 方針明確: 設計方針は明確だが、具体的なデータ構造やAPI定義が必要
△ 追加設計必要: 概念は定義されているが、実装に必要な詳細が不足
× 未定義: 計画書にほぼ記述がなく、ゼロからの設計が必要
```

## 2. 対話セッションにあり計画書に未反映の情報

対話の中で議論されたが `planning_doc.md` に明示的に含まれていない要素。
詳細計画の策定前に計画書へ反映するか、別文書として分離すべきもの。

| 情報 | 対話での議論内容 | 計画書での状態 | 推奨対応 |
|---|---|---|---|
| キャンペーン構造 | サイクル連鎖、インターリュードの5バリエーション、キャンペーン終了条件 | 言及はあるが体系的な記述なし | 計画書にセクション追加 or 別文書 |
| 読者アクターの四象限の起承転結パターン | 起承転結の各段階での理想的分布が詳細に議論された | 計画書に含まれていない | 計画書の付録として追加 |
| 小規模物語への適用 | スケール不変性、心理的情報遮断、アクター構成例 | 付録D でスケール定義は反映済み。小規模の具体例は未反映 | 付録D に補足追加 or 別文書 |
| Author の「無知の執筆者」設計の詳細根拠 | AUT が HO-PRV を見ないことの文学的・実装的理由 | セクション3.1 で遮断ルールとしてのみ記載 | 設計判断記録（ADR）として別文書 |
| EDT のフィードバックフィルタリング | EVL-RPT から創作改善への変換方法 | セクション5.2 で「EDT が参照」のみ | フィルタリングルールの設計が必要 |

## 3. 実装フェーズの提案

### フェーズ0: 基盤整備（計画書の補完）

```

  □ 全 JSON Schema の確定（ST-CHR, ST-ORG, ST-NAT, ST-ENV, EVL-*）
  □ 全プロンプトテンプレートの初版作成（CHR, AUT, EDT, ENV, NAT, ORG, RDR）
  □ キャンペーン構造の計画書への反映
  □ 四象限パターンの計画書への反映

  □ 遅延再分類アルゴリズムの設計

推定規模: 計画書の追記約500行 + JSON Schema 10-15ファイル
```

```
  - ENV   : 1（簡易版。シーン開始時の天候メモのみ）

  - 1サイクル（4シーン。起承転結 各1シーン）
```

## 4. 総合評価

### 4.1 「詳細実装計画を作成できるか」への回答

**回答: 条件付きでYes。**

```
現在の準備状況:
  ◎（実装可能）  : 12 コンポーネント
  ○（方針明確）  : 22 コンポーネント
  △（追加設計必要）:  7 コンポーネント
  ×（未定義）    :  0 コンポーネント
```

**×（未定義）がゼロ**であることが最も重要な結果である。
対話セッションと計画書を合わせると、実装に必要な概念は全て定義されている。

**◎ と ○ で全体の83%を占める**ことから、大半のコンポーネントは
JSON Schema の確定とプロンプトテンプレートの作成を行えば
詳細実装計画の策定が可能である。

**△ の7コンポーネント**はいずれも「数値化・アルゴリズム化」が必要な
評価系の要素（緊張度曲線、層間コントラスト、遅延再分類等）である。
これらは概念は明確だが、具体的な算出式が未定義。
フェーズ0で設計を完了させる必要がある。

### 4.2 計画書の強みと弱み

**強み:**

```
- 情報可視性の設計が網羅的かつ厳密
  → 全アクター × 全データストアのマトリクスが完成している
  → 条件付きアクセスのルールが具体的
  → 遮断の厳格性が3レベルで定義されている
  
- 情報フロー（シーン生成パイプライン）が実装可能なレベルで定義されている
  → STEP1-6 の各STEPで入力/出力が明記
  → 「何を見せない」が各STEPで明記

- タイムスケール設計が実用的
  → 活性/休眠/待機の3状態モデル
  → 典型的な稼働パターンの図示
  → 休眠↔活性の遷移ルールが明確

- Git管理との統合が具体的
  → コミット戦略、hook仕様、ブランチ戦略が運用レベルで定義

- スケール不変性が論証されている
  → スケール1-5の定義と遷移パターンにより、
    同一アーキテクチャでの多様な物語への適用が保証されている
```

**弱み（詳細計画前に解決が必要）:**

```
- 評価系の数値化が未完了
  → 四象限の概念は明確だが、スコア算出の具体式がない
  → 緊張度曲線、層間コントラストの測定方法が未定義
  → 遅延再分類のアルゴリズムが未定義

- プロンプトテンプレートが未作成
  → 設計上の入出力は定義されているが、
    実際の LLM に渡すプロンプト文面が存在しない
  → これは最も工数のかかる作業であり、
    品質に直結する（プロンプトの質 = 出力の質）

- キャンペーン構造が計画書に未統合
  → 対話中に詳細に議論されたが、計画書への反映が部分的
  → インターリュードの5バリエーションは計画書に概略のみ

- 人間（HMN）インターフェースの未設計
  → HMN の権限は定義されているが、
    「人間が実際にどう操作するか」のUI/UXが未考慮
  → CLI? Web UI? 対話型? 
  → これは実装方針に大きく影響する
```

### 4.3 最大のリスク

```
リスク1: EDT への過負荷
  EDT は全データを参照し、全判断を行う。
  プロンプトのコンテキスト長が爆発する可能性がある。
  → 緩和策: EDT の入力を段階的に構成する設計が必要。
    全データを毎回渡すのではなく、判断フェーズに応じて
    参照データを切り替える。

リスク2: プロンプトテンプレートの品質
  システムの出力品質は最終的にプロンプトの質に依存する。
  計画書がいかに精密でも、プロンプトが貧弱なら結果は貧弱。
  → 緩和策: フェーズ1（MVP）で早期にプロンプトの
    試行錯誤サイクルを回し、計画書の設計と
    プロンプトの現実を擦り合わせる。

リスク3: APIコスト
  全アクターの稼働 × シーン数 × サイクル数 × 並行世界数。
  計算上、1サイクル12シーンで数百回のAPI呼び出しになりうる。
  → 緩和策: タイムスケール設計（セクション4）による
    アクター休眠の厳格な運用。不必要なアクターは動かさない。
```

## 5. 結論とネクストアクション

### 詳細実装計画の策定に進むために必要な作業（優先順）

```
優先度1（詳細計画のブロッカー）:
  □ 全アクターのプロンプトテンプレート初版を作成
  □ 全データストアの JSON Schema を確定
  □ EDT のコンテキスト管理戦略を設計（過負荷対策）

優先度2（フェーズ0に含めるべき）:
  □ 評価系の算出アルゴリズム設計（四象限スコア、緊張度、層間コントラスト）
  □ 遅延再分類エンジンのアルゴリズム設計
  □ キャンペーン構造の計画書への統合
  □ 起承転結パターンの理想分布を計画書に反映

優先度3（フェーズ1開始前に決定すべき）:
  □ HMN インターフェースの方式決定（CLI / Web UI / 対話型）
  □ 使用する LLM の選定と制約の確認（コンテキスト長、コスト）
  □ Git リポジトリの初期セットアップスクリプト
```

### 一文での要約

**対話セッションと計画書により、本システムの「何を作るか」は十分に定義されている。「どう作るか」の詳細化——特にプロンプトテンプレート、JSON Schema、評価アルゴリズム——をフェーズ0として実行すれば、詳細実装計画の策定に進むことができる。**
