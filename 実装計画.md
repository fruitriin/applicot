# Applicot — 実装計画

`planning_doc.md`（設計仕様）と `プロジェクトゴール.md`（準備状況評価）を入力として、
**Bun ランタイム上で動作するマルチエージェント小説執筆システムの具体的な実装計画**を定義する。

---

## アーキテクチャ方針

### ランタイムとツールチェイン

| 項目 | 選定 | 理由 |
|---|---|---|
| ランタイム | **Bun** | 高速起動、TypeScript ネイティブ、ファイルI/O高速 |
| 言語 | **TypeScript** (strict) | 型安全性によるアクター間データ流通の検証 |
| パッケージマネージャ | **bun install** | Bun 統合 |
| テスト | **bun:test** | 追加依存なし |
| LLM クライアント | **Anthropic SDK** (`@anthropic-ai/sdk`) | Claude を主LLMとして想定。他LLM対応はアダプタ層で吸収 |

### ストレージ方針: ファイルシステムファースト

**全データをファイルシステム上の JSON / Markdown で管理する。** データベースは使用しない。

```
設計判断の根拠:
  ✅ git diff で全変更を人間が読める形で追跡可能
  ✅ git blame で「誰が（どのアクターが）いつ変更したか」を追跡可能
  ✅ ブランチ/ワークツリーで並行世界が自然に分離される
  ✅ cherry-pick/merge の標準 Git 操作がそのまま物語操作になる
  ✅ バックアップ = git push
  ⚠️ パフォーマンスはファイルI/Oに依存（許容する）
  ⚠️ 大規模データの検索はインメモリインデックスで補完
```

### Git リポジトリ分離

**ツール（applicot）と物語データ（novel）は別の Git リポジトリとして管理する。**
相互にバージョンを参照しない。サブモジュール化しない。

```
~/workspace/
├── applicot/              # Git リポジトリ A: ツール本体
│   ├── .git/
│   ├── src/
│   ├── tests/
│   └── package.json
│
└── my-novel/              # Git リポジトリ B: 物語データ
    ├── .git/
    ├── handouts/
    ├── state/
    ├── scenes/
    ├── recall/
    └── ...

接続方法:
  applicot --novel-dir ../my-novel scene next
  または APPLICOT_NOVEL_DIR=../my-novel を環境変数で指定
```

```
設計判断の根拠:
  ✅ ツールのバージョンアップが物語データに影響しない
  ✅ 物語データだけをバックアップ・共有できる
  ✅ 複数の物語プロジェクトで同一ツールを使い回せる
  ✅ 物語データの git log が「物語の歴史」だけを記録する
  ✅ ツールのコミット履歴と物語のコミット履歴が混在しない
```

### プロジェクト構成

```
applicot/                  # ← Git リポジトリ A
├── src/
│   ├── core/              # コアドメイン

│   └── cli/               # CLIインターフェース
│       └── index.ts       # エントリーポイント

│   ├── state/

│   ├── recall/
```

## 未決定事項と設計リスク

### フェーズ0着手前に決定が必要

| 項目 | 選択肢 | 推奨 |
|---|---|---|
| プロンプトテンプレートエンジン | Handlebars / Mustache / 自前文字列テンプレート | 自前。依存を減らし、可視性フィルタとの統合を容易にする |
| LLM の選定 | Claude Sonnet / Opus / 混合 | アクター種別で使い分け。EDT=Opus, CHR/AUT=Sonnet, RDR=Haiku |
| recall の要約に使う LLM | 同一 / 別（軽量モデル） | Haiku で要約。コスト抑制 |

### 設計リスクと緩和策

| リスク | 影響 | 緩和策 |
|---|---|---|
| EDT のコンテキスト過負荷 | プロンプトが長すぎて品質低下 | フェーズ別入力分離。サマリ生成の導入 |
| プロンプト品質 | システム出力品質に直結 | フェーズ1で早期に試行錯誤。テンプレートのイテレーション |
| API コスト | 全アクター×シーン数で数百回/サイクル | タイムスケール設計の厳格運用。不必要なアクターは動かさない |
| ファイルI/Oのボトルネック | 大量ファイルの読み書きで遅延 | Bun の高速ファイルI/O。必要に応じてインメモリキャッシュ |
| Git リポジトリ肥大化 | JSON の差分が大きくなる | JSON のキー順序を固定（deterministic stringify）。状態ファイルのエンティティ単位分割 |
